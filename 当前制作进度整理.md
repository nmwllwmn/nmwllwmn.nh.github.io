# 《最后一次签到》当前制作进度与下一阶段规划

更新时间：2026-06-11  
当前仓库：`nmwllwmn/nmwllwmn.nh.github.io`  
当前方向：以“捡到林舟的手机”为核心入口的网页解谜游戏。
协作约定：后续代码或页面更新完成并通过基础检查后，默认提交并推送到 GitHub，保持线上网页实时同步。

## 1. 当前整体判断

项目已经从“多个网页拼线索”的雏形，推进到“手机内调查”为主的版本。现在玩家的主要行为是打开林舟手机，在微信、短信、相册、邮箱、浏览器、地图、电话、备忘录、相机、数独等 App 之间寻找线索。

当前已经具备可玩的核心框架：

- 主页 `index.html` 可以进入南洄。
- 手机主界面 `chat.html` 已有锁屏、实时钟、上滑解锁、解锁状态保存、桌面 App、控制中心、通知中心。
- 手机 App 多数已经拆成独立页面，符合“每次打开新页面/新窗口”的方向。
- 音乐功能、顶部灵动岛/灰色胶囊、外部旁白提醒已经移除或隐藏。
- 手机端不再依赖外部笔记浮窗，线索主要来自手机内部。
- SEO/GEO 文件已经加入，Google 验证已完成。

目前最大问题不是“功能数量不够”，而是“体验深度、逻辑闭合和最终流程验收还不够”。下一阶段应该优先把现有 App 做扎实，而不是继续无序新增页面。

## 4. 当前结局结构

`archive.html` 已有三个方向：

### 普通结局：最后一次签到

条件：核心证据齐全，但缺少救援路线。  
含义：玩家知道林舟没有真正完赛，也知道后半程被人补写，但没找到真正入口。

### 真相结局：旧站未删除

条件：核心证据齐全，并完成数独彩蛋。  
含义：数独恢复出匿名号码与后台账号的联系，玩家选择备份证据并报警，但林舟仍未归队。

### 隐藏结局：未完成的签到

条件：核心证据齐全，同时找到 B17、泵站入口、00:17 维护窗口。  
含义：玩家拼出救援路线，找到林舟。S08 仍显示未完成。

当前结局框架是对的，但还需要加强“玩家如何自然走到 archive.html”和“每个结局差异是否足够有冲击力”。

## 5. 当前主要问题

### 5.1 手机真实感仍需补强

虽然页面已拆分，但部分 App 仍像“信息面板”，不像完整手机应用。优先需要补强：

- 微信消息页的未读红点、读后消失、返回逻辑。
- 短信会话列表和不同短信线程，而不是只围绕一条威胁短信。
- 电话 App 的拨号、未接记录、详情页。
- 设置 App 的应用锁、设备信息、登录账号、备份时间。

### 5.2 主线流程还缺明确“玩家路线”

现在很多线索能点出来，但玩家可能不知道下一步去哪。需要设计一条清晰但不明说的主线：

1. 微信看到“北桥不是终点”和 `0606`。
2. 备忘录确认邮箱账号、密码和骚扰短信。
3. 短信发现 21:41 威胁。
4. 相机进入相册锁，用 `062141` 解锁。
5. 相册看到 B17、许听雨、泵站照片。
6. 浏览器历史查旧站、S08、密页集。
7. 邮箱确认 Web Import 和 NVR 入口，不直接暴露执行账号。
8. 地图搜索 `IMP-2202`，确认执行账号。
9. 地图定位 B17 和泵站。
10. 进入密页集判断结局。

### 5.3 40 层调查进度没有完整分配

`common.js` 已设置 `MAX_STAGE = 40`，但当前各页面 stage 更像粗略标记。后面要把 40 层拆成 40 个明确调查节点，每一层对应一个可见动作或发现。

当前手机电量不再直接等于 stage，而是主线线索进度。普通结局主线按 100% 分配：解锁手机 4%、读阿远微信 8%、队伍/导入提示 4%、威胁短信 4%、相册锁 6%、证据 A/B/C 各 12%、邮箱登录 6%、NVR 入口 6%、证据 D 12%、证据 E 14%。证据 A-E 齐全时电量归 0%，触发普通结局条件；未齐全时最低保留 1%，避免误判完成。

### 5.4 NVR/监控系统还没有手机内完整体验

目前证据 D 指向 21:38 北桥监控异常，但手机内还缺一个完整的监控/邮件附件/NVR 登录体验。可以不做成大平台，但至少要在手机里形成闭环。

### 5.5 外部旧页面与手机主体验关系要重新定位

`event.html`、`university.html`、`club2023.html`、`nvr.html`、`map.html`、`mail.html`、`wall.html` 等旧式网页仍存在。后续应决定：

- 要么改为手机浏览器里的伪网页内容；
- 要么保留为“从手机浏览器打开的新窗口”；
- 不能让玩家感觉突然离开手机主玩法。

## 6. 下一阶段规划

### 第一阶段：闭合主线流程

目标：玩家不看攻略也能从手机自然推进到普通结局。

要做：

1. 把 40 层调查进度表补齐到文档和代码。
2. 逐页校准时间线，消除微信、短信、电话、邮箱、相册之间的矛盾。
3. 给所有密码补足来源提示，尤其是相册锁、邮箱、密页集入口。
4. 强化 `archive.html` 的入口，让玩家从浏览器历史或相册/邮箱自然打开。
5. 验证普通结局完整路径。

验收：

- 从首页开始，第一次游玩可以完成普通结局。
- 不依赖作者说明，不需要猜随机密码。
- 所有关键证据都有明确页面来源。

### 第二阶段：手机 App 质感升级

目标：把当前“能用”的 App 做成“像真的”。

要做：

1. 微信：消息列表、聊天详情、通讯录、朋友圈返回关系统一；未读红点读后消失。
2. 短信：增加真实短信线程、验证码、运营商、未知号码详情；威胁短信作为自然混入。
3. 电话：增加未接电话详情，区分系统电话和微信语音。
4. 设置：加入应用锁、设备信息、账号、备份记录，为相册锁和邮箱提供二次线索。
5. 相册：照片详情视觉化，增加破损照片预览、EXIF、隐藏照片层级。

验收：

- 每个 App 有明确的进入、返回、详情、反馈。
- 页面不再像简单文本列表。
- 手机中心位置和页面切换稳定，不需要玩家每次重新调整视角。

### 第三阶段：结局与隐藏内容

目标：普通结局、数独彩蛋结局、隐藏救援结局都能被明确触发。

要做：

1. 普通结局：证据齐全但救援路线不完整。
2. 数独彩蛋结局：完成三关数独后恢复额外后台绑定记录。
3. 隐藏结局：必须同时找到 B17、泵站、00:17 维护窗口。
4. 给三个结局写更强的最后文本和视觉反馈。
5. 防止结局条件互相覆盖或误触发。

验收：

- 三个结局条件互斥或优先级清晰。
- 数独是彩蛋，不阻断普通主线。
- 隐藏结局需要额外探索，但线索可推导。

### 第四阶段：旧页面整合与 SEO/GEO 收尾

目标：让作品既好玩，也能被搜索和 AI 摘要正确理解。

要做：

1. 检查 `sitemap.xml` 是否包含所有有效页面。
2. 保持 `robots.txt`、`llms.txt`、`ai-summary.json`、`site.webmanifest`、Google 验证文件不被删除。
3. 给新增页面补足标题、描述、Open Graph。
4. 决定旧网页是并入手机浏览器，还是作为可打开的新窗口保留。
5. 上传 GitHub，保持线上同步。

验收：

- 页面无 404。
- Google 验证文件仍可访问。
- sitemap 页面与实际页面一致。
- AI/GEO 摘要能准确描述作品类型、玩法、作者。

## 7. 建议立刻执行的下一步

下一步不建议再新增大功能。建议按这个顺序做：

1. 制作“40 层调查进度表”，把每层对应页面、触发动作、证据/flag 写清楚。
2. 按这张表校准 `common.js` 的 stage 和 evidence。
3. 重做微信、短信、电话三个最常用 App 的真实感和返回逻辑。
4. 补一个手机内 NVR/监控证据闭环，确保 D 证据不是空来的。
5. 完整跑一遍普通结局、数独结局、隐藏结局。

这会比继续堆页面更有效。现在作品已经有骨架，下一步要做的是把骨架上的每个关节拧紧。
